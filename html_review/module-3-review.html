3.1 — Configure Virtual Networks

Learning Objectives

  • Describe Azure virtual network features and components
  • Identify features and usage cases for subnets and subnetting
  • Identify usage cases for private and public IP addresses
  • Create a virtual network and assign an IP address

What is an Azure Virtual Network?

A logical isolation of Azure cloud resources. Each VNet has its own CIDR block and can be linked to other VNets and on-premises networks (when CIDR blocks don't overlap).

Key Characteristics

  • You control DNS server settings and subnet segmentation
  • Can be used for VPNs, hybrid connections, and dedicated private cloud networks
  • Resources in different VNets cannot communicate by default — must use peering or VPN
  • Subnet ranges within a VNet cannot overlap each other

Subnets

Subdivide a VNet's address space for better security and performance. Azure reserves the first 4 and last 1 IP addresses in each subnet for internal use.

  • Use subnets to segment and isolate workloads (e.g., web tier vs. data tier)
  • Network Security Groups and route tables can be applied at the subnet level

IP Addresses

Public IP

  • For internet-facing resources and connecting to Azure public-facing services
  • Dynamic — can change when VM is stopped/deallocated (common source of connectivity issues after restart)
  • Static — required for DNS servers, VPN gateways, app gateways, etc.

Private IP

  • For internal communication; not exposed to internet
  • Use when a VM needs to communicate with on-premises without internet exposure
  • Assigned from VNet address space

Use Static IPs When:

  • Setting up DNS servers
  • VPN gateways and Azure Application Gateways
  • Resources that require a consistent IP for firewall rules or DNS resolution

Connecting VNets to On-Premises

  • Site-to-Site VPN — encrypted connection over the internet between on-premises and Azure VNet
  • Address spaces must not overlap between on-premises and Azure VNet

3.2 — Configure Network Security Groups

Learning Objectives

  • Determine when to use network security groups
  • Create network security groups
  • Implement and evaluate network security group rules
  • Describe the function of application security groups

What is an NSG?

A list of security rules that allow or deny inbound or outbound network traffic to/from Azure resources. Can be associated with a subnet or a network interface (NIC).

NSG Rules

  • Each rule has: priority, source/destination (IP, CIDR, service tag, ASG), protocol (TCP/UDP/Any), port range, and action (Allow/Deny)
  • Lower priority number = higher precedence (rule 100 is processed before rule 200)
  • A new rule with a higher priority value (e.g., 500) is processed AFTER rules with lower values (e.g., 100)
  • Default rules exist at priority 65000–65500 (allow VNet/Azure LB traffic, deny all inbound/outbound)

Effective Security Rules

Azure aggregates rules from the subnet NSG and the NIC NSG. Use the Effective security rules view (Network Watcher) to see the combined rules that apply to a NIC.

Application Security Groups (ASGs)

Logical groupings of VMs by application role (e.g., "WebServers", "DBServers"). Instead of specifying IPs in NSG rules, you reference ASG names — this makes rules maintainable as IP addresses change.

Key Benefits

  • Define network security based on application logic rather than IP addresses
  • Simplify security rule management for workloads with minimal complexity
  • Group VMs by function; apply rules to the group

DMZ Pattern

A demilitarized zone (DMZ) acts as a buffer between internal resources and external traffic — use NSG rules to restrict what can pass through.

Use NSGs When:

You need to restrict traffic flow to resources in a virtual network based on specific security rules. Recommended for securing VMs and Azure service networking.

3.3 — Host a Domain with Azure DNS

Learning Objectives

  • Configure Azure DNS to host your domain

What is Azure DNS?

Azure DNS lets you host your DNS zones on Azure infrastructure using the same credentials, APIs, tools, and billing as other Azure services. Azure DNS is a hosting service for DNS domains — it resolves domain names to IP addresses.

Key Facts

  • You cannot register domain names with Azure DNS — use a third-party domain registrar, then delegate to Azure DNS
  • Delegation: update registrar's name server records to point to Azure DNS name servers
  • Supports private DNS zones for name resolution within virtual networks
  • Supports alias records (ALIAS / CNAME at zone apex) to point to Azure public IP addresses, Traffic Manager profiles, or CDN endpoints
  • Highly available — backed by anycast routing to nearest name server cluster

Common DNS Record Types

  • A — maps hostname to IPv4 address
  • AAAA — maps hostname to IPv6 address
  • CNAME — alias for another hostname; cannot be used at zone apex
  • MX — mail exchange records
  • NS — name server records (used for delegation)
  • TXT — text records (used for domain verification, SPF)
  • Alias record (ALIAS) — Azure-specific; can point zone apex to Azure resources

3.4 — Configure VNet Peering

Learning Objectives

  • Identify usage cases and product features of Azure VNet peering
  • Configure VPN Gateway for transit connectivity
  • Extend peering using hub-and-spoke with UDRs and service chaining

What is VNet Peering?

Connects two Azure virtual networks so resources in both can communicate. Traffic between peered VNets is kept on the Microsoft backbone network (not the public internet). Can connect VNets in the same or different regions.

Peering Status

  • When successfully peered, the status for both VNets shows "Connected"
  • Peering is non-transitive — if A↔B and B↔C, A cannot reach C unless A↔C is also peered
  • Address spaces of peered VNets must not overlap

Gateway Transit

Allows a peered VNet to use the VPN Gateway of the hub VNet to connect to on-premises or other remote networks. The spoke VNet enables "Use remote gateways"; the hub VNet enables "Allow gateway transit".

Hub-and-Spoke Topology

Hub VNet contains shared services (gateway, firewall, DNS). Spoke VNets peer to the hub. User-Defined Routes (UDRs) and service chaining route traffic through hub appliances.

3.5 — Control Network Traffic Flow with Routes

Learning Objectives

  • Identify the routing capabilities of an Azure virtual network
  • Configure routing within a virtual network
  • Deploy a basic network virtual appliance
  • Configure routing to send traffic through a network virtual appliance

Azure Route Types

  • System routes — automatically created by Azure; handle traffic within VNets, to the internet, and between peered VNets. Cannot be deleted but can be overridden.
  • User-Defined Routes (UDRs) — custom routes in a route table that you create and associate with subnets. Used to override system routes or force traffic through a network virtual appliance (NVA).
  • BGP routes — propagated from on-premises networks via VPN Gateway or ExpressRoute

Network Virtual Appliances (NVAs)

A VM that performs network functions (firewall, WAF, routing). Traffic is redirected through an NVA using UDRs so it can be inspected before being allowed through.

  • Next hop type in a UDR can be: Virtual appliance, Virtual network gateway, VNet, Internet, None (drop)
  • Enable IP forwarding on the NVA's NIC so it can route packets

3.6 — Intro to Azure Load Balancer

Learning Objectives

  • Learn what Azure Load Balancer is and the functionality it provides
  • Determine whether Load Balancer meets the needs of your organization

What is Azure Load Balancer?

Distributes inbound TCP/UDP network traffic across a group of Azure VMs or VM Scale Set instances. Operates at OSI Layer 4 (Transport layer). Supports millions of TCP and UDP flows.

Public vs Internal Load Balancers

  • Public Load Balancer — maps public IP/port to private IP/port of backend VMs; load balances internet traffic; also provides outbound connections for backend VMs
  • Internal (Private) Load Balancer — front-end never exposed to internet; used for internal LOB apps, multi-tier backends, cross-premises VPN traffic

Key Components

  • Health probes — ensure traffic isn't sent to unhealthy backend instances (e.g., probe TCP port 443 for HTTPS)
  • Load-balancing rules — define how traffic is distributed across the backend pool
  • Session persistence (sticky sessions) — ensures all requests from the same client go to the same backend server; required when a rule has a source IP/port hash applied
  • Inbound NAT rules — allow RDP/SSH to individual backend VMs through the load balancer's public IP using different ports

3.7 — Intro to Azure Application Gateway

Learning Objectives

  • Learn what Azure Application Gateway is and the functionality it provides
  • Determine whether Application Gateway meets the needs of your organization

What is Azure Application Gateway?

A web traffic load balancer operating at OSI Layer 7 (Application layer). Routes HTTP/HTTPS traffic based on URL, hostname, or path. Includes a built-in Web Application Firewall (WAF).

Key Features

  • Web Application Firewall (WAF) — protects against SQL injection, XSS, and other OWASP vulnerabilities
  • TLS/SSL termination — decrypts traffic at the gateway to reduce CPU load on backend servers; can re-encrypt to backend (end-to-end encryption)
  • Session affinity (stickiness) — routes a client's requests to the same backend server (important for e-commerce sessions)
  • Health probes — stops forwarding to unresponsive backend servers automatically
  • Connection draining — allows existing connections to complete naturally before removing a backend member (for planned maintenance)
  • Autoscaling — dynamically adjusts capacity based on traffic load
  • Supports HTTP, HTTPS, HTTP/2, and WebSocket

3.8 — Intro to Azure Network Watcher

Learning Objectives

  • Learn what Azure Network Watcher is and the functionality it provides
  • Determine whether Network Watcher meets the needs of your organization

What is Azure Network Watcher?

A suite of tools to monitor, diagnose, and log for Azure IaaS resources. It is not designed for PaaS or Web analytics.

Tool Categories

Monitoring

  • Topology — visual map of all resources and their relationships across subscriptions/resource groups; useful for understanding a complex network at the start of troubleshooting
  • Connection monitor — continuous end-to-end connectivity monitoring between endpoints; monitors over time (unlike Connection troubleshoot which is point-in-time)

Network Diagnostic Tools

  • IP flow verify — checks if a specific packet (src/dst IP, port, protocol, direction) is allowed or denied by NSG rules; identifies which rule is responsible
  • NSG diagnostics — like IP flow verify but works at VM, VMSS, or Application Gateway level; can suggest new rules
  • Next hop — detects routing issues; shows next hop type, IP, and route table ID for a destination
  • Effective security rules — shows combined NSG rules from subnet + NIC
  • Connection troubleshoot — tests connectivity at a point in time (compare to Connection monitor which is continuous)
  • Packet capture — remotely capture all network traffic to/from a VM
  • VPN troubleshoot — diagnoses virtual network gateway connections; returns health diagnosis

Traffic

  • Flow logs — logs Azure IP traffic info; stores in Azure Storage; works with NSGs or VNets
  • Traffic analytics — rich visualizations of flow logs data

When NOT to Use Network Watcher

  • Diagnosing PaaS services — check Azure status/Service Health dashboard instead
  • Advanced diagnostics requiring third-party tools
  • Web analytics

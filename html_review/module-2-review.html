2.1 — Understand Microsoft Entra ID

What is Microsoft Entra ID?

Microsoft Entra ID (formerly Azure Active Directory) is a cloud-based identity and access management service. It is primarily designed for internet-based applications using HTTP (port 80) and HTTPS (port 443). It is a multi-tenant directory service.

Entra ID vs Active Directory Domain Services (AD DS)

FeatureAD DSMicrosoft Entra ID
StructureHierarchical X.500Flat (no OUs/GPOs)
AuthenticationKerberosSAML, WS-Federation, OpenID Connect
AuthorizationKerberos ticketsOAuth
QueryingLDAPREST API over HTTP/HTTPS

GPOsYesNo
DNSUsed for resource locationNot required

Microsoft Entra ID License Tiers

P1 — includes:

  • Self-service group management
  • Advanced security reports & alerts (ML-based)
  • Multi-factor authentication (MFA)
  • Microsoft Identity Manager (MIM) licensing for hybrid identity
  • Enterprise SLA of 99.9%
  • Password reset with writeback to on-premises AD
  • Cloud App Discovery
  • Conditional Access based on device, group, or location
  • Microsoft Entra Connect Health

P2 — adds (on top of P1):

  • Microsoft Entra ID Protection — user risk policies, sign-in policies, behavior monitoring
  • Microsoft Entra Privileged Identity Management (PIM) — permanent & temporary admins, policy workflow for privileged access

Key Facts

  • Entra ID uses OAuth for authorization (not Kerberos)
  • Identity protection feature monitors for irregular sign-in activities
  • Entra ID OU structure is flat — no Group Policy Objects (GPOs)
  • Microsoft Entra Connect Health: operational insight into identity infrastructure
  • MIM bridges on-premises stores (AD DS, LDAP, Oracle) to Entra ID

2.2 — Create, Configure & Manage Identities

User Types in Microsoft Entra ID

  • Cloud identities — exist only in Entra ID; deleted when removed from primary directory
  • Directory-synchronized identities — originate in on-premises AD; synced via Microsoft Entra Cloud Sync (recommended for most) or Microsoft Entra Connect Sync (complex scenarios like device sync or groups >50,000 members)
  • Guest users — external accounts (other cloud providers, MSA); source is "Invited user"

Group Types

Security Groups

  • Manage access to shared resources
  • Members: users, devices, service principals
  • Requires a Microsoft Entra administrator

Microsoft 365 Groups

  • Collaboration: shared mailbox, calendar, files, SharePoint
  • Available to users and admins
  • Can include external users

Group Membership Types

  • Assigned — manually added/maintained
  • Dynamic User — automatically managed based on user attributes (department, job title, location); requires Entra ID P1
  • Dynamic Device — same but for devices; security groups only (not M365 groups)

Group-based Licensing

  • Assign licenses to a group; all members receive them automatically
  • Requires Entra ID P1 or higher (or Office 365 E3+)
  • You need at least as many licenses as unique members in licensed groups
  • License modifications effective within minutes of membership change
  • If same license from multiple sources, consumed only once
  • Managed via Microsoft 365 admin center
  • Always set usage location on user profiles before assigning licenses

Common License Error States

  • CountViolation — not enough available licenses; increase license count
  • MutuallyExclusiveViolation — conflicting licenses/service plans; remove conflicting items
  • DependencyViolation — dependent service plans must be reassigned before removing a license

2.3 — Core Azure Architectural Components

Physical Infrastructure

  • Datacenters — not accessed directly; grouped into regions and availability zones
  • Regions — geographical area with one or more low-latency-linked datacenters; you choose a region when deploying resources
  • Availability Zones — physically separate datacenters within a region with independent power/cooling/networking; minimum 3 zones in AZ-enabled regions
  • Region Pairs — most regions paired with another region ≥300 miles away within the same geography; enables cross-region replication, prioritized recovery, staged updates

Availability Zone Service Categories

  • Zonal — pinned to a specific zone (VMs, managed disks, IP addresses)
  • Zone-redundant — auto-replicates across zones (zone-redundant storage, SQL Database)
  • Non-regional — globally resilient, not tied to a zone (Entra ID, Traffic Manager, DNS)

Region Pair Key Facts

  • Planned updates rolled out one region at a time
  • Data residency remains within geography (except Brazil South → South Central US)
  • Brazil South: one-directional pair; South Central US's secondary is NOT Brazil South
  • Some newer regions (Italy North, Poland Central, Israel Central) use AZs + GRS instead of pairs

Management Infrastructure

Hierarchy (top to bottom): Tenant Root Group → Management Groups → Subscriptions → Resource Groups → Resources

  • Resources — VMs, VNets, DBs, etc.; each belongs to exactly one resource group; actions on a group affect all its resources
  • Resource Groups — cannot be nested; cannot be renamed after creation; deleting a group deletes everything in it
  • Subscriptions — unit of management, billing, and scale; two boundary types: Billing boundary (separate invoices) and Access control boundary (separate policies & spending limits)
  • Management Groups — organize subscriptions; policies applied at MG level inherit down; max 10,000 MGs per directory; max 6 nesting levels (not counting root/subscription)

2.4 — Azure Policy

What is Azure Policy?

Azure Policy assesses compliance at scale and enforces organizational and regulatory standards across Azure environments. It operates in the control plane and evaluates resources against policy rules. Policies can be assigned at Management Group, Subscription, or Resource Group level.

Policy vs RBAC

In the ARM request pipeline, RBAC is checked before Azure Policy. If a user lacks permissions (RBAC fails), Policy is never evaluated.

Compliance Scanning

  • Greenfield (policy-first) — policy evaluated at resource creation/update time
  • Brownfield (resource-first) — existing resources evaluated by compliance scan; runs automatically every 24 hours or can be triggered manually; non-compliant resources are flagged but not deleted

Policy Effects (then block)

EffectBehaviorEval Type
denyBlocks non-compliant resource requestsSync
auditLogs a warning in activity log; doesn't block the requestAsync
modifyAdds, updates, or removes properties/tags on resourcesSync
deployIfNotExistsRuns a template deployment when condition is metAsync
auditIfNotExistsAudits related resources missing required propertiesAsync
disabledDeactivates the policy assignment; checked firstSync
appendAdds fields to resource (mostly obsolete; use modify)Sync
manualSelf-attest compliance state via custom attestationsManual

Enforcement Mode

  • Use Enforcement Mode Disabled when testing policies — evaluates compliance without applying the policy effect
  • Recommended rollout: start with disabled enforcement, then deploy in deployment rings

Policy Logical Operators (if block)

  • allOf — all conditions must be true (logical AND)
  • anyOf — one or more conditions must be true (logical OR)
  • not — inverts the result of a condition

2.5 — Secure Azure Resources with RBAC

What is Azure RBAC?

An authorization system built on Azure Resource Manager that provides fine-grained access management. You control access through role assignments — binding a role to a security principal at a specific scope.

Role Assignment = Who + What + Where

  • Security Principal (who) — user, group, service principal, or managed identity
  • Role Definition (what) — a collection of permissions (read, write, delete); can be built-in or custom
  • Scope (where) — Management Group → Subscription → Resource Group → Resource (child scopes inherit from parent)

Built-in Roles (4 fundamental)

  • Owner — full access to all resources + can delegate access to others
  • Contributor — can create and manage all resource types; cannot grant access to others
  • Reader — can view existing resources; no write/delete
  • User Access Administrator — can manage user access to Azure resources

RBAC is an Allow Model

Permissions are additive — multiple role assignments stack. The NotActions property creates exclusions from the Actions wildcard. Effective permissions = Actions minus NotActions.

Key Operations

  • View who has access: Access control (IAM) pane in Azure portal
  • Check a specific user's access: resource/group → Access control (IAM) → Check Access
  • Review role assignments history: Activity log filtered on Create role assignment (roleAssignments) operation
  • Least-privilege: assign roles at the narrowest scope that satisfies the requirement

2.6 — Self-Service Password Reset (SSPR)

What is SSPR?

Allows users to reset their own passwords without calling the help desk. Requires Microsoft Entra ID P1 or P2 (or Microsoft 365 Apps for business) for users who are not already signed in.

SSPR Authentication Methods

Six available methods; administrators choose which to enable (recommendation: enable ≥2):

  • Mobile app notification (Authenticator)
  • Mobile app code (Authenticator)
  • Email (external address)
  • Mobile phone (SMS or automated call)
  • Office phone (automated call)
  • Security questions — least recommended; never use alone; not available for admin accounts

Key Rules

  • Admin accounts always require a strong two-method policy regardless of tenant config
  • Security questions are NOT available for admin accounts
  • Mobile phone (SMS) is not a recommended primary method due to SMS fraud risk
  • A user is "registered for SSPR" once they've enrolled the required minimum number of methods
  • Password writeback to on-premises AD requires Entra ID P1/P2 or M365 Apps for business

SSPR Rollout Scopes

  • None — default; SSPR disabled for all users
  • Selected — enabled for a specific security group (use for pilot testing)
  • All — enabled for all users in the tenant

Notifications

  • Notify users on password resets — alerts the user via primary/secondary email
  • Notify all admins when other admins reset their password — cross-admin awareness

Password Writeback Deployment Options

  • Microsoft Entra Connect — existing on-premises AD users; complex scenarios
  • Cloud Sync — higher availability (no single agent dependency); supports disconnected domains (e.g., post-merger); can run side-by-side with Connect

4.1 — Configure Storage Accounts

Azure Storage Services

  • Blob Storage — unstructured data (objects); supports block blobs, append blobs, page blobs
  • Azure Files — managed file shares (SMB/NFS)
  • Queue Storage — reliable messaging between app components
  • Table Storage — NoSQL structured data (key-value)
  • Azure Disk Storage — persistent block storage for IaaS VMs (managed disks)

Securing Storage Endpoints

  • Restrict access to specific subnets via Firewalls and virtual networks settings in the storage account
  • Private Endpoints — assign a private IP from your VNet to the storage account (traffic stays off public internet)
  • Enable Default to Microsoft Entra authorization to prefer RBAC over shared keys
  • Use Azure Key Vault to manage and rotate storage account access keys

4.2 — Configure Blob Storage

Blob Containers

  • All blobs must be in a container; a storage account can have unlimited containers
  • Container names: lowercase letters, numbers, hyphens; 3–63 characters; must start with letter or number
  • Public access levels:
    • Private (default) — no anonymous access
    • Blob — anonymous read access to blobs only
    • Container — anonymous read + list access to entire container
  • Blob/Container access levels only take effect if Allow Blob anonymous access is enabled at the account level

Blob Access Tiers

TierLatencyMin DurationUse Case
HotmsNoneFrequently accessed data; highest storage cost, lowest access cost
Coolms30 daysInfrequent access; short-term backup, older media
Coldms90 daysRarely accessed; lower cost than Cool
ArchiveHours180 daysOffline; compliance/backup data; cheapest storage, expensive access

Rehydrating from Archive: Copy Blob (recommended, creates new online blob) or Set Blob Tier. Priority options: Standard (≤15 hrs) or High priority (≤1 hr for <10 GB).

Blob Lifecycle Management

Rule-based policy using If-Then conditions to automatically transition blobs between tiers or delete them based on age/access patterns.

  • Transition direction: Hot → Cool → Cold → Archive (one way, except Cool → Hot on access)
  • Delete current/previous versions or snapshots at end of lifecycle
  • Apply rules to entire account, specific containers, or subsets by name prefix or blob index tags

Object Replication

  • Asynchronously replicates block blobs between storage accounts (same or different regions)
  • Requires blob versioning enabled on both source and destination accounts
  • Does NOT support snapshot replication

4.3 — Configure Storage Security

Authorization Options (in order of preference)

  • Microsoft Entra ID + RBAC — recommended primary method; use role assignments
  • Shared Access Signatures (SAS) — time-limited, scoped tokens for delegated access
  • Storage account access keys — full account access; manage with Azure Key Vault

Shared Access Signatures (SAS)

A URI that grants restricted, time-limited access to Azure Storage resources without sharing account keys.

SAS Types

  • User delegation SAS — secured with Entra ID credentials; supported for Blob + Data Lake Storage; most secure
  • Service-level SAS — grants access to specific resources; can be backed by a stored access policy
  • Account-level SAS — broader access (including service-level capabilities + account-level operations)
  • Stored access policy — server-side policy that groups SAS tokens; allows revocation without regenerating keys

SAS Best Practices

  • Always use HTTPS for creating and distributing SAS tokens
  • Use stored access policies to enable revocation without key rotation
  • Set near-term expiry times for unplanned SAS tokens
  • Set start time at least 15 minutes in the past to handle clock skew
  • Grant minimum required permissions only
  • Use SAS for read access for a limited time (e.g., temporary image sharing)

Storage Encryption

  • All data encrypted at rest by default using Storage Service Encryption (SSE) with AES-256
  • Customer-managed keys — store keys in Azure Key Vault instead of using Microsoft-managed keys
  • Data encrypted in transit by default; always ensure HTTPS is enabled (enforce "Secure transfer required")

4.4 — Configure Azure Files & File Sync

Azure Files vs Blob Storage

Azure Files

  • True directory object hierarchy
  • SMB and NFS protocols
  • Ideal for lift-and-shift, shared tools, diagnostic data
  • Up to 100 TiB per share, 4 TiB per file

Blob Storage

  • Flat namespace (container + blob)
  • REST API, HTTP access
  • Ideal for streaming, random-access, app data accessible from anywhere
  • Massively scalable

File Share Snapshots

  • Point-in-time, read-only copy of the file share
  • Incremental — only changes since the last snapshot are saved (reduces cost)
  • Integrates with File Explorer's "Previous Versions" feature
  • Use for recovery from accidental deletions, failed upgrades, ransomware protection

Soft Delete for Azure Files

  • Enabled at the storage account level
  • Deleted file shares transition to a soft-deleted state instead of permanent deletion
  • Retention period: 1–365 days
  • Configure retention to align with organizational data retention requirements

Azure File Sync

Replicates Azure file shares to Windows Servers (on-premises or cloud). Enables distributed caching and performance for data at the location where it's used.

  • Supports multi-site access to the same data
  • Tiering: frequently accessed files cached locally; infrequently accessed files tiered to Azure
  • Use for replacing DFS (Distributed File System) or centralizing file share management

Identity-Based Access for Azure Files

  • Supports Microsoft Entra ID or AD DS identities synced to Entra ID
  • Accessing Azure Files through Azure Storage Explorer requires Microsoft Entra ID permissions

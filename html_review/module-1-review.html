1.1 — Intro to Azure Cloud Shell

Learning Objectives

  • Describe Azure Cloud Shell and the functionality it provides
  • Determine whether Cloud Shell meets the needs of your organization
  • Recognize how to use Cloud Shell and persist files for multiple sessions

What is Azure Cloud Shell?

Azure Cloud Shell is a browser-accessible command-line environment for managing Azure resources. It supports both Bash and PowerShell. Because Microsoft manages Cloud Shell, you always have access to the latest versions of the Azure CLI and PowerShell modules without installing anything locally.

Key Facts

  • Accessible from any browser — no local installation required
  • Infrastructure is compliant with double encryption at rest by default
  • Sessions terminate after 20 minutes of inactivity (state is lost; CloudDrive files persist)
  • Only one instance is allowed at a time — not suitable for concurrent multi-tenant work

How to Access Cloud Shell

  • Direct link: shell.azure.com
  • Via the Azure portal (icon in top navigation bar)
  • From code snippets in Microsoft Learn

Persisting Files — CloudDrive

Cloud Shell allocates a temporary VM per session. To persist files across sessions, use the Azure CloudDrive (backed by Azure File Share). Files on CloudDrive survive session termination and are accessible from any device. You can also map an Azure Storage File Share to access its contents through Cloud Shell.

Use the Cloud Shell editor (curly-brackets icon, or code <filename>) to edit files directly in CloudDrive. The code command requires Classic mode.

Pre-installed Tools

Cloud Shell sessions include: Azure CLI, AzCopy, Azure Functions CLI, Docker Machine, kubectl, Helm, Terraform, Ansible, git, vim, and more — no manual installation needed.

When to Use (and When NOT To)

✓ Use Cloud Shell when…

  • Accessing from any browser-based device
  • No local plug-ins/add-ons available
  • Need to persist files across sessions
  • Prefer Bash or PowerShell flexibility
  • Editing scripts via built-in editor

✗ Avoid Cloud Shell when…

  • Sessions longer than 20 minutes (will disconnect)
  • Need sudo / admin permissions
  • Need tools not supported in Cloud Shell
  • Storage from multiple regions is required
  • Multiple concurrent sessions needed

1.2 — Create ARM Templates with VS Code

Learning Objectives

  • Implement a JSON ARM template using Visual Studio Code
  • Declare resources and add flexibility with parameters and outputs

What is an ARM Template?

ARM templates are JSON files that define the infrastructure and configuration for your Azure deployment using declarative syntax — you describe what to deploy, not how to deploy it step by step. Azure Resource Manager then deploys resources in the correct order, in parallel where possible.

Note: Bicep is now the recommended language for new IaC on Azure (simpler syntax than JSON ARM templates).

Benefits of ARM Templates

  • Idempotent — deploy the same template multiple times; only changes are applied
  • Consistent configurations — same template = same outcome every time
  • Parallel resource creation — faster than scripted deployments
  • Built-in validation — Resource Manager checks the template before deployment starts
  • CI/CD integration — works with Azure Pipelines and GitHub Actions
  • Deployment history — visible in the Azure portal with parameter/output values

ARM Template File Structure

ElementRequired?Purpose
$schemaRequiredLocation of JSON schema file describing template structure
contentVersionRequiredTemplate version (e.g., 1.0.0.0)
parametersOptionalValues provided at deployment time; max 256
variablesOptionalValues to simplify template expressions
functionsOptionalUser-defined functions for reuse in the template
resourcesRequiredThe actual resources to deploy or update
outputsOptionalValues returned after successful deployment
apiProfileOptionalCollection of API versions for resource types

Parameters

Parameters let you customize deployments for different environments. Best practices:

  • Use parameters for environment-specific settings (SKU, size, capacity, names)
  • Never hardcode or provide defaults for passwords/secrets — always use parameters of type secureString or secureObject
  • Provide a description for each parameter
  • Use default values where appropriate
  • Limit 256 parameters per template

Parameter types: string, secureString, int, bool, object, secureObject, array

Reference a parameter with: [parameters('parameterName')]

Outputs

Outputs return values after deployment (e.g., a storage account's endpoint). Use the reference() function to get runtime state of a deployed resource.

Deploying an ARM Template

Deploy locally using Azure CLI or Azure PowerShell:

  • Azure CLI: az deployment group create --template-file azuredeploy.json --resource-group myRG
  • Use az deployment group create (preferred) instead of the deprecated az group deployment create
  • Can pass parameter values inline: --parameters storageAccountType=Standard_LRS

Resource type syntax: {resource-provider}/{resource-type} (e.g., Microsoft.Storage/storageAccounts)

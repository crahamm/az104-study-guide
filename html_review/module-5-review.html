5.1 — Intro to Azure Virtual Machines

VM Creation Checklist

  • The network (VNet, subnets, overlapping address spaces)
  • VM name (up to 64 chars Linux / 15 chars Windows; used as computer name)
  • Location/region (affects available sizes, pricing, compliance)
  • VM size (CPU, memory, disk trade-offs)
  • Disks (OS disk + data disks; separate data from OS)
  • Operating system (affects licensing cost)

VM Size Families

  • General purpose — balanced CPU/memory; testing, dev, small DBs, web servers
  • Compute optimized — high CPU-to-memory; medium web servers, batch, app servers
  • Memory optimized — high memory-to-CPU; relational DBs, large caches, in-memory analytics
  • Storage optimized — high disk throughput/IOPS; databases
  • GPU — graphics rendering, video editing, deep learning model training/inference
  • High performance compute — fastest CPUs, optional high-throughput networking

VM Billing Model

  • Compute — billed per minute; stops when VM is stopped and deallocated (not just stopped). Includes OS license for Windows. Cheaper for Linux.
  • Storage — billed separately; charged even when VM is stopped/deallocated
  • Reserved VM Instances (RI) — 1 or 3-year commitment; up to 72% savings vs pay-as-you-go
  • Azure Hybrid Benefit — reuse on-premises Windows/Linux licenses on Azure VMs

5.2 — Configure VM Availability

Availability Sets

Group VMs within a datacenter across fault domains and update domains to avoid single points of failure.

  • Fault domains (FD) — VMs in different FDs share no common power or network switch; default 2 FDs, max 3
  • Update domains (UD) — VMs in different UDs are not rebooted simultaneously during planned maintenance; default 5 UDs, up to 20
  • Provides 99.95% SLA when ≥2 VMs in an availability set
  • Must be set at VM creation — cannot add to an existing availability set later
  • All VMs in an availability set are within a single datacenter (same region); no protection against datacenter failure

Availability Zones

Deploy VMs to physically separate datacenters (different power, cooling, networking) within a region. Provides 99.99% SLA. Protects against datacenter-level failures.

Virtual Machine Scale Sets (VMSS)

Create and manage a group of identical, load-balanced VMs that can automatically scale based on demand or a schedule.

  • All instances use the same base OS image and configuration
  • Integrates with Azure Load Balancer or Application Gateway
  • Supports autoscaling with scale-out (add VMs) and scale-in (remove VMs) based on rules
  • Scale up = increase VM size (more CPU/RAM) — no extra VMs added
  • Scale out = add more VM instances (horizontal scaling)

Autoscale Rules

  • Metric-based — scale based on CPU %, memory %, queue length, etc.
  • Time-based (schedule) — scale at specific times; e.g., scale out at 8:00 AM on Saturdays

5.3 — Configure App Service Plans

What is an App Service Plan?

Defines the compute resources for web apps to run on. All apps in the same plan share the same VM instances. The plan is the scale unit for App Service — if you scale the plan, all apps in it scale together.

Pricing Tiers (3 Categories)

Shared Compute — Dev/Test only; NO SLA

  • Free (F1) & Shared (D1) — run on same VMs as other customers' apps; CPU quotas only; no scale-out

Dedicated Compute — Production workloads

  • Basic (B) — dedicated VMs, manual scaling up to 3 instances; no autoscale, no staging slots
  • Standard (S) — auto scale, up to 10 instances, 5 staging slots, 10 daily backups
  • Premium (P) / PremiumV3 (P1V3) — higher performance, up to 30 instances, 20 staging slots; recommended for new production deployments

Isolated — Network + compute isolation; max scale

  • Isolated / IsolatedV2 — dedicated VNet; up to 200 instances; for mission-critical, compliance-sensitive workloads

Scale Up vs Scale Out

  • Scale up — increase plan tier for more CPU, memory, disk; no additional VMs
  • Scale out — add more VM instances running the same app

Triggering a scale event at 8:00 AM on Saturdays = time-based (schedule) rule.

5.4 — Configure Azure App Services

What is Azure App Service?

Fully managed PaaS platform for hosting web apps, REST APIs, and mobile backends. Supports Windows and Linux. Languages: ASP.NET, Java, Node.js, Python, PHP, Ruby, .NET.

Deployment Slots

Live apps with their own hostnames; content and config can be swapped between slots (including production).

  • Available in Standard, Premium, and Isolated tiers (not Free or Basic)
  • Standard: 5 slots; Premium/Isolated: 20 slots
  • Swap — swap staging ↔ production with zero downtime; all instances in the destination slot are warm before traffic redirects
  • Auto swap — deploy to slot → automatically swaps to production when warmed up; for CI/CD zero-downtime workflows
  • Restore last known good — swap back immediately if a deployment is bad

Sticky vs Non-sticky Settings

  • Sticky (slot-specific) — stays with the slot when swapped: connection strings (when marked slot-specific), app settings marked as slot-specific
  • Non-sticky (follows content) — moves with the swap: connection strings (by default), general app settings

Exam tip: Connection strings follow the content across a swap.

Automated Deployment Sources

  • GitHub, Azure DevOps, Bitbucket, Docker Hub, Azure Container Registry

Monitoring: Application Insights

Provides detailed telemetry about your web app — page usage, traffic by time of day, user location, performance metrics. Use for Marketing/analytics questions about which pages are popular and when.

Backup

  • Available in Standard and above tiers
  • Can back up app + database; store in Azure Storage

Azure Container Instances (ACI)

Fastest and simplest way to run a container in Azure without managing VMs. Great for isolated, short-lived container workloads.

  • Start in seconds — no VM provisioning needed
  • Expose directly to internet with public IP and FQDN
  • Custom CPU (0.1–4 vCPU) and memory (0.1–16 GB) per container group
  • Supports Linux and Windows containers
  • Mount Azure Files file shares for persistent storage
  • Linux containers can be deployed into a VNet for private communication

Container Groups

The top-level resource in ACI. Like a Kubernetes pod — a collection of containers scheduled on the same host, sharing lifecycle, network, and storage.

  • Containers in a group share a public IP and expose ports (port mapping not supported — containers share port namespace)
  • Deploy multi-container groups using: ARM template (JSON), Bicep (recommended), or YAML files
  • Resources for the group = sum of all containers' resource requests
  • Use cases: web app + content puller, app + logging sidecar, front-end + back-end, monitoring + app

Container Images

A container image is a lightweight, portable, executable package containing code, runtime, system tools, libraries, and settings. First step to deploy ACI = create a container image using Docker.

Deploy Multiple Containers

Use an orchestrator such as Azure Kubernetes Service (AKS) to deploy and manage multiple containers.

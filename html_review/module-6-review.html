6.1 — Intro to Azure Backup

What is Azure Backup?

An Azure service that provides zero-infrastructure, secure, cost-effective backup solutions for all Azure-managed data assets. Managed from a central console (Backup Center).

What Can Be Backed Up

  • Azure VMs (Windows and Linux)
  • Azure Managed Disks
  • Azure File Shares
  • SQL Server and SAP HANA databases in Azure VMs
  • Azure Blobs, Azure Database for PostgreSQL/MySQL
  • Azure Kubernetes clusters
  • On-premises files, folders, system state (via MARS agent)

Key Concepts

RTO and RPO

  • Recovery Time Objective (RTO) — maximum tolerable downtime after a disaster
  • Recovery Point Objective (RPO) — maximum tolerable data loss measured in time; determines backup frequency

Azure Backup vs Azure Site Recovery

FeatureAzure BackupAzure Site Recovery
Primary goalMaintain point-in-time copies; go back in timeNear-real-time replication; enable failover
Use caseAccidental deletion, data corruption, ransomwareRegion-wide disasters, business continuity

Backup Storage Tiers

  • Snapshot tier — stored locally for up to 5 days; fastest recovery (instant restore); retained in customer's subscription
  • Vault-standard tier — online storage in Recovery Services vault; isolated copy even if source is deleted; GRS recommended for vaults
  • Archive tier — long-term retention (LTR) for compliance; rarely accessed data at lowest cost

Security Features

  • Soft delete — backup data retained for 14 additional days after deletion; protects against accidental/malicious deletion; free of charge during retention period
  • Enhanced soft delete — longer soft-delete period; can be set to "always on" (cannot be turned off)
  • RBAC — restrict backup/restore operations to authorized Backup Admins
  • Data encrypted in transit and at rest; supports customer-managed keys (CMK) in Key Vault
  • Azure VM backup uses Azure backbone network — no internet connectivity required

Management Tools

  • Recovery Services vault — stores backup data; defines policies; acts as RBAC boundary
  • Backup Center — single pane of glass across multiple workloads, vaults, subscriptions, regions, and Azure Lighthouse tenants
  • Backup Explorer — aggregated view of entire backup estate with drill-down troubleshooting

Backup Replication Options for Vaults

  • LRS — lowest cost; basic protection; noncritical scenarios
  • GRS — recommended for backup scenarios; cross-region restore capability
  • ZRS — protects against datacenter-level failures; high-availability scenarios

6.2 — Protect Virtual Machines with Azure Backup

VM Backup Process

  1. Configure backup policy (frequency, retention)
  2. At first backup: Azure installs a backup extension on the running VM
    • Windows: VM Snapshot extension (uses VSS — Volume Shadow Copy Service)
    • Linux: VM SnapshotLinux extension (disk snapshot)
  3. Snapshot is taken → stored locally (snapshot tier) → transferred to vault
  4. Subsequent backups transfer only changed blocks (delta/incremental)

Prerequisite to back up a VM: Backup extensions must be installed.

Backup Types

  • Full — entire database; max once per day (daily or weekly interval)
  • Differential — changes since last full backup; max once per day (not same day as full)
  • Transaction Log — point-in-time recovery up to a second; configurable every 15 minutes (SQL)
  • Incremental/hourly VM — Enhanced policy allows hourly VM backups (4, 6, 8, 12, or 24 hr intervals); min RPO of 4 hours
  • Selective disk backup — back up a subset of disks to reduce cost

VM Snapshot Consistency Levels

  • Application consistent — captures memory and pending I/O (VSS on Windows; pre/post scripts on Linux); best option for databases
  • File system consistent — consistent snapshot if VSS/scripts fail; apps may need cleanup on startup
  • Crash consistent — VM was shut down during backup; no I/O or memory captured; least reliable

Restore Options

  • Create a new VM — creates a basic VM from a restore point; must be in same region as source
  • Restore disk — restores disk to a resource group; use to customize VM config or attach to existing VM
  • Replace existing — replaces disk on existing VM (VM must still exist; not available for encrypted VMs)
  • Cross-region restore — restore to Azure paired secondary region; supported for Create VM and Restore Disks
  • Cross-subscription restore — restore to different subscription in same tenant; requires CRR enabled on vault
  • File-level recovery — mount snapshot via iSCSI initiator to recover individual files (not supported for encrypted VMs — must restore full VM)

Instant Restore

Restores from the snapshot tier (local) — faster than vault tier. Snapshots retained locally for up to 5 days. After 5 days, use vault tier restore.

6.3 — Monitor Azure VMs Using Diagnostic Data

Azure Monitor Overview

Comprehensive monitoring solution for collecting, analyzing, and responding to monitoring data from Azure resources. Two main data types:

  • Metrics — numerical values at defined intervals (e.g., CPU %, disk ops/sec); retained for 93 days; stored in Azure Monitor Metrics
  • Logs — events with timestamps; stored in Log Analytics workspace; queried with KQL (Kusto Query Language)

VM Monitoring Layers

  1. Host VM — compute, storage, network allocated by Azure
  2. Guest OS — the operating system running on the VM
  3. Client workloads — services/processes on the OS
  4. Applications — apps running on the VM

Host VM Metrics (automatically collected)

Azure automatically collects these for every VM:

  • VM availability
  • CPU usage % (average)
  • OS disk usage (total)
  • Network operations (total)
  • Disk operations per second (average)

Use Azure Monitor Metrics Explorer to plot, compare, and investigate metrics. Use Recommended alert rules for predefined CPU/memory/disk/network/availability thresholds.

Activity Logs

Automatically recorded for all Azure resources (VM startup, configuration changes, etc.). Can be sent to:

  • Azure Monitor Logs — complex querying, alerts, up to 2-year retention
  • Azure Storage — cheap long-term archiving
  • Azure Event Hubs — forward outside Azure

Boot Diagnostics

Host logs for troubleshooting boot issues. Enable at VM creation or afterwards. Provides screenshot from hypervisor and serial console log output (Linux). Stored in a managed storage account.

Guest OS and Application Monitoring

Requires installing Azure Monitor Agent and creating a Data Collection Rule (DCR).

  • DCR defines what data to collect and where to send it
  • Azure Monitor Metrics — stores metrics (performance counters) only
  • Azure Monitor Logs — stores both metrics AND event logs

VM Insights

Azure Monitor feature that simplifies guest OS monitoring with simplified onboarding, a preconfigured DCR, predefined performance charts and workbooks, and optionally a dependency map of processes and external connections.

Use when: you don't know which metric to focus on yet; exploring overall usage and performance.
